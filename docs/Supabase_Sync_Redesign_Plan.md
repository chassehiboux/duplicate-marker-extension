# План переработки Supabase-синхронизации

Источник требований: `docs/TZ_Codex_Supabase_Sync_Redesign.md`.

## Пользовательское поведение

После входа в Supabase-аккаунт актуальные данные пользователя хранятся в Supabase. Локальное хранилище браузера используется как быстрый кэш, очередь неотправленных изменений и техническое состояние текущего устройства.

Если пользователь входит во второй браузер, локальные старые данные второго браузера не отправляются поверх облака автоматически. Расширение сначала получает состояние из Supabase и заменяет им локальный кэш. Перед заменой сохраняется локальная резервная копия.

Если пользователь меняет настройку, счетчик, тег или правило действия, интерфейс может обновиться сразу. Изменение попадает в локальную очередь и отправляется в Supabase как mutation. Если сеть недоступна, изменение остается в очереди и будет отправлено позже.

Если в другом браузере появилось изменение, Realtime сообщает только о новой серверной ревизии. После этого расширение делает pull через REST/RPC и применяет подтвержденное сервером состояние.

Кнопка "Синхронизировать сейчас" выполняет безопасный цикл: обновляет сессию, регистрирует устройство, сначала отправляет локальную очередь, затем подтягивает canonical state. Она не должна затирать локальные неотправленные изменения.

Импорт JSON без входа остается локальным. После входа импорт требует подтверждения и отправляет импортируемые sync-данные в Supabase как batch mutation, затем применяет canonical state.

## Изменения UI

Блок Supabase остается в верхней части popup.

В свернутом виде показывается:

- режим: "Supabase включен" или "Синхронизация";
- email пользователя или "Локальное хранение";
- визуальный индикатор ошибки, если есть проблема pull/push или очередь не пуста.

В раскрытом виде показывается отдельная диагностика:

```text
Состояние:
Realtime: подключен
Последний pull: 14.05 12:10
Последний push: 14.05 12:09
Server revision: 128
Local revision: 128
Очередь изменений: 0
Последняя ошибка: -
```

При проблемах статус не должен писать "все синхронизировано", если есть очередь или REST-ошибка:

```text
Realtime: подключен
REST push: ошибка NetworkError
Очередь изменений: 2
Данные будут отправлены повторно автоматически
```

## Техническая схема

Старый active-write путь `collectLocalSyncSnapshot() -> upsertRemoteState() -> extension_user_state.state` отключается. Таблица `extension_user_state` остается только legacy-источником для миграции и диагностики.

Новые таблицы:

- `extension_devices`;
- `extension_sync_state`;
- `extension_user_kv`;
- `extension_counter_days`;
- `extension_processed_edits`;
- `extension_processed_edocids`;
- `extension_action_rules`;
- `extension_mutations`.

Дополнительная таблица `extension_processed_edocids` нужна потому, что текущий синхронизируемый ключ `processed_edocids` отвечает за защиту счетчика анализа исполнения от повторного учета документов. Если оставить его обычным JSON в key-value, два браузера смогут потерять часть отметок при одновременной работе.

RPC:

- `extension_register_device`;
- `extension_get_state`;
- `extension_apply_mutation`.

Локальные technical keys:

- `dup_supabase_device_id_v1`;
- `dup_supabase_outbox_v1`;
- `dup_supabase_cache_revision_v1`;
- `dup_supabase_sync_meta_v2`;
- `dup_supabase_pre_login_backup_v2`.

## Область изменений

Scope gate:

- StageTimer: нет;
- Firefox-совместимость: да, потому что меняются background/content/popup и deploy-состав может быть затронут;
- `manifest.json`, permissions, content scripts, background scripts, ресурсы, deploy-скрипты: возможно, только если sync API будет вынесен в отдельный файл;
- новые элементы UI/F2/скриншот: нет;
- новые постоянные ключи `chrome.storage.local`: да, все `dup_supabase_*` классифицируются как технические и не синхронизируются;
- Supabase-синхронизация: да;
- Apps Script / Google Sheets bridge: да, только в части перехода sync-ключа bridge URL на единый sync API.

## Миграция данных

Legacy данные из `extension_user_state.state` не удаляются. При первом переходе пользовательские данные переносятся в новую схему.

Если у пользователя уже есть данные в новой схеме с `revision > 0`, локальный браузер не делает bootstrap поверх Supabase.

Если новая схема пуста, текущие локальные sync-ключи отправляются как bootstrap mutation.

## Проверка

Минимальная проверка после реализации:

- первый вход с локальными данными;
- вход второго браузера с отличающимися локальными данными;
- изменение настройки в одном браузере и получение ее другим через Realtime + pull;
- NetworkError при push с сохранением outbox;
- конфликт ревизий;
- одновременный increment счетчика из двух браузеров;
- Realtime подключен, REST pull/push упал;
- отсутствие активного full snapshot upsert.
