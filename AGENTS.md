# Локальные правила для Codex

## Архитектура двух расширений

1. В проекте есть два варианта расширения:
   - основное расширение, корневой `manifest.json`: использует `background.js`, `content.js`, а также подключает модуль StageTimer (`StageTimer/timer_content.js` + `StageTimer/timer_styles.css`);
   - обособленное расширение StageTimer: собирается и деплоится через `deploy_VZID.py`, использует `StageTimer/timer_background.js` как `service_worker` и подключает `StageTimer/timer_content.js` + `StageTimer/timer_styles.css`.
2. Любые изменения, влияющие на StageTimer, вносить синхронно для обеих сборок:
   - логика таймера и UI — `StageTimer/timer_content.js`;
   - стили таймера — `StageTimer/timer_styles.css`;
   - фоновая логика StageTimer — `StageTimer/timer_background.js` и соответствующая секция `background.js` основного расширения.
3. Если правка затрагивает подключение файлов, состав сборки или поведение обособленного расширения, проверять и при необходимости обновлять `deploy_VZID.py`.
4. Нельзя оставлять функциональное расхождение между основным и обособленным расширением без явной пометки и причины в итоговом сообщении.
5. Если изменение расширения затрагивает состав файлов, `manifest.json`, permissions, background/content scripts, ресурсы, StageTimer или Firefox-совместимость, проверять и при необходимости дорабатывать затронутые деплои:
   - `deploy.py`;
   - `deploy_VZID.py`;
   - `deploy-firefox.py`.
6. Не оставлять ситуацию, когда код изменился, а сборочный или деплойный скрипт больше не собирает актуальный состав расширения.

## Scope gate перед проверками

1. Перед массовыми проверками один раз определить затронутые области:
   - StageTimer: да/нет;
   - Firefox-совместимость: да/нет;
   - `manifest.json`, permissions, content scripts, background scripts, ресурсы, deploy-скрипты: да/нет;
   - новые элементы UI, меню `F2`, селекторы скрытия/блокировки при скриншоте: да/нет;
   - новые постоянные ключи `chrome.storage.local` или долговечные ключи `window.localStorage`: да/нет;
   - Supabase-синхронизация: да/нет;
   - Apps Script bridge / Google Sheets bridge: да/нет.
2. Если область отмечена как “нет” и после этого не было изменений, влияющих на неё, не возвращаться к её проверкам повторно.
3. Если в ходе работы появилась новая правка, которая меняет scope, обновить scope gate и проверить только вновь затронутые области.
4. Не запускать широкие проверки “на всякий случай”, если есть понятный меньший набор файлов и областей, который реально изменился.

## Автоскрытие всех элементов расширения при создании скриншота

1. Все существующие элементы расширения должны скрываться при создании скриншота.
2. При создании новых элементов необходимо удостовериться, что они будут скрываться при создании скриншота.
3. При добавлении новых элементов расширения или удалении старых обновлять корневой `extension_ui_config.js`, состав пунктов меню настроек по `F2` и связанные селекторы скрытия/блокировки для основного расширения и StageTimer, если правка их затрагивает.

## Supabase-синхронизация постоянных данных

Если новый или изменяемый функционал сохраняет постоянные пользовательские данные локально, агент обязан сразу решить вопрос их Supabase-синхронизации в этой же задаче.

Данные нельзя оставлять только локальными, если они должны сохраняться после переустановки, переноситься между браузерами или быть доступны после входа в Supabase-аккаунт.

При необходимости агент сам изучает текущую Supabase-реализацию проекта, вносит нужные изменения в код и БД, оформляет изменения через миграции и выполняет все действия с БД через Supabase MCP.

Пользователя нельзя просить вручную выполнять SQL в Supabase.

В итоговом сообщении агент должен явно указать, что сделано с новыми локальными данными:
- добавлены в Supabase-синхронизацию;
- либо оставлены только локальными с кратким объяснением причины.
