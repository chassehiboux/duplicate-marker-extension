# Локальные правила для Codex

## Кодировка, русский текст и безопасное редактирование

1. Все текстовые файлы с кириллицей сохранять в UTF-8 без BOM, если формат файла явно не требует другого.
2. Русский пользовательский текст всегда оставлять на русском языке, не переводить на английский без прямой просьбы пользователя.
3. Не заменять русский текст на `?????`, `����`, mojibake-последовательности или `\uXXXX`-экранирование вместо обычной кириллицы в строках интерфейса.
4. Не выполнять отдельные проверки кодировки после каждого редактирования. Вместо этого всегда молча использовать безопасный способ чтения и записи файлов.
5. Предпочтительный способ внесения правок — `apply_patch` или встроенные средства редактирования Codex/VS Code, не зависящие от кодировки терминала.
6. Если нужен скрипт для чтения или записи файлов, использовать Python с явной кодировкой:

   ```python
   from pathlib import Path

   path = Path("file.js")
   text = path.read_text(encoding="utf-8")
   path.write_text(text, encoding="utf-8", newline="")
   ```

7. Не использовать Windows PowerShell 5.1-команды для чтения, перезаписи или массовой замены текста в проектных файлах, если файл может содержать кириллицу:
   - `Get-Content`;
   - `Set-Content`;
   - `Add-Content`;
   - `Out-File`;
   - `-replace` с последующей записью;
   - любые пайплайны, которые читают и затем перезаписывают файл через PowerShell.
8. Визуальный вывод терминала, `cat`, `type`, `Get-Content` или IDE не считать доказательством корректной или некорректной кодировки.
9. Не выполнять перекодировку, “починку UTF-8” или массовую перезапись файлов без явного признака реальной проблемы в байтах файла.
10. Если обнаружены реальные маркеры уже существующего mojibake в изменяемом тексте, исправлять только затронутый фрагмент и не переписывать файл целиком без необходимости.
11. При финальной проверке перед коммитом UTF-8/mojibake-проверку выполнять только для staged-файлов с русским текстом, если они менялись в текущей задаче.

## StageJump и журнал действий

1. При задачах по StageJump/автофильтру обязательно вести пошаговый журнал в `STAGEJUMP_DEBUG_LOG.md`.
2. Перед каждым новым действием по StageJump/автофильтру сверяться с последним зафиксированным шагом в `STAGEJUMP_DEBUG_LOG.md`.
3. Не дублировать уже выполненные шаги, если журнал подтверждает, что после них не было изменений, влияющих на текущий вывод.

## Архитектура двух расширений

1. В проекте есть два варианта расширения:
   - основное расширение, корневой `manifest.json`: использует `background.js`, `content.js`, а также подключает модуль StageTimer (`StageTimer/timer_content.js` + `StageTimer/timer_styles.css`);
   - обособленное расширение StageTimer: собирается и деплоится через `deploy_VZID.py`, использует `StageTimer/timer_background.js` как `service_worker` и подключает `StageTimer/timer_content.js` + `StageTimer/timer_styles.css`.
2. Любые изменения, влияющие на StageTimer, вносить синхронно для обеих сборок:
   - логика таймера и UI — `StageTimer/timer_content.js`;
   - стили таймера — `StageTimer/timer_styles.css`;
   - фоновая логика StageTimer — `StageTimer/timer_background.js` и соответствующая секция `background.js` основного расширения.
3. Если правка затрагивает подключение файлов, состав сборки или поведение обособленного расширения, проверять и при необходимости обновлять `deploy_VZID.py`.
4. Нельзя оставлять функциональное расхождение между основным и обособленным расширением без явной пометки и причины в итоговом сообщении.
5. Если изменение расширения затрагивает состав файлов, `manifest.json`, permissions, background/content scripts, ресурсы, StageTimer или Firefox-совместимость, проверять и при необходимости дорабатывать затронутые деплои:
   - `deploy.py`;
   - `deploy_VZID.py`;
   - `deploy-firefox.py`.
6. Не оставлять ситуацию, когда код изменился, а сборочный или деплойный скрипт больше не собирает актуальный состав расширения.

## Scope gate перед проверками

1. Перед массовыми проверками один раз определить затронутые области:
   - StageTimer: да/нет;
   - Firefox-совместимость: да/нет;
   - `manifest.json`, permissions, content scripts, background scripts, ресурсы, deploy-скрипты: да/нет;
   - новые элементы UI, меню `F2`, селекторы скрытия/блокировки при скриншоте: да/нет;
   - новые постоянные ключи `chrome.storage.local` или долговечные ключи `window.localStorage`: да/нет;
   - Supabase-синхронизация: да/нет;
   - Apps Script bridge / Google Sheets bridge: да/нет.
2. Если область отмечена как “нет” и после этого не было изменений, влияющих на неё, не возвращаться к её проверкам повторно.
3. Если в ходе работы появилась новая правка, которая меняет scope, обновить scope gate и проверить только вновь затронутые области.
4. Не запускать широкие проверки “на всякий случай”, если есть понятный меньший набор файлов и областей, который реально изменился.

## Экономия проверок и кэш результатов

1. Не повторять одну и ту же проверку для файла, если после последнего успешного результата этот файл не изменялся.
   - Пример: если `node --check GoogleSheets/problem_picker_content.js` уже прошёл, повторять его только после новых изменений в этом файле или в финальном pre-commit наборе.
2. Если проверка уже прошла и затем изменился только один файл, повторять только проверки, применимые к этому файлу.
3. Не запускать заново весь набор JS/GS/deploy-проверок без причины.
4. Проверки кодировки не использовать как рутинный шаг после каждого редактирования. Главная защита от mojibake — безопасный способ чтения и записи файлов.
5. Если UTF-8/mojibake-проверка всё же нужна, она должна быть точечной и проверять только файлы, реально затронутые текущей задачей.
6. Каноническая UTF-8/mojibake-проверка должна проверять:
   - валидное UTF-8-декодирование байтов;
   - отсутствие явных маркеров искажения: `Рџ`, `РЎ`, `Р’`, `Ð`, `Â`, `����`, `????`;
   - только файлы, затронутые текущей задачей.
7. Длинные PowerShell one-liner-команды для UTF-8/mojibake не писать вручную.
8. Если inline-команда сломалась из-за quoting/escaping, не повторять её в похожем виде; перейти на сохранённый скрипт или Python с явным `encoding="utf-8"`.
9. Делать не больше трёх git-проверок состава за задачу без отдельной причины:
   - в начале: зафиксировать уже существующие dirty-файлы;
   - перед staging: проверить, что в `git add` попадают только файлы задачи;
   - перед коммитом: `git diff --cached --check` и финальный `git status --short`.
10. `git diff --check` и `git diff --cached --check` запускать по смыслу:
   - `git diff --check` — перед staging, если надо поймать whitespace в рабочем дереве;
   - `git diff --cached --check` — перед коммитом;
   - не повторять обе команды после каждой мелкой правки, если изменился только один уже проверяемый файл.
11. Web/Context7/документацию использовать только когда нужна новая информация.
12. Если официальный источник уже подтвердил поведение и не появилось противоречий, не искать то же самое повторно.
13. Для браузерных различий Firefox/Chrome предпочтительно использовать официальную документацию MDN/Chrome/WebExtensions и один раз зафиксировать вывод.
14. Если проверка была пропущена из-за кэша успешного результата, в итоговом сообщении можно кратко указать: “не повторялась, файл после успешной проверки не менялся”.

## Автоскрытие всех элементов расширения при создании скриншота

1. Все существующие элементы расширения должны скрываться при создании скриншота.
2. При создании новых элементов необходимо удостовериться, что они будут скрываться при создании скриншота.
3. При добавлении новых элементов расширения или удалении старых обновлять корневой `extension_ui_config.js`, состав пунктов меню настроек по `F2` и связанные селекторы скрытия/блокировки для основного расширения и StageTimer, если правка их затрагивает.

## Supabase-синхронизация постоянных данных

1. Любой новый постоянный ключ `chrome.storage.local` или долговечный ключ `window.localStorage` в основном расширении должен быть явно классифицирован в `supabase_sync.js`.
2. Новый ключ нужно либо добавить в синхронизацию, если он должен переживать переустановку или работать между устройствами, либо добавить в исключения с понятной причиной.
3. Временные, технические и привязанные к конкретному устройству данные не синхронизировать:
   - логи;
   - runtime/session-состояния;
   - cookieStore/container id;
   - временные StageJump payload;
   - данные StageTimer, которые относятся к отправке в Google Sheets.
4. Если новый ключ хранится только в `window.localStorage`, но должен синхронизироваться, добавить мост в `chrome.storage.local`, потому что Supabase-синхронизация работает через `chrome.storage.local`.

## Финальная проверка перед коммитом

1. Перед коммитом проверять только staged-файлы и реально затронутые области из scope gate.
2. Обязательно выполнить:
   - `git diff --cached --check`;
   - синтаксические проверки для staged JS/GS-файлов, если они менялись;
   - точечную UTF-8/mojibake-проверку для staged-файлов с русским текстом, если они менялись в текущей задаче;
   - проверку состава staged-файлов, чтобы не включить чужие изменения.
3. Не включать в коммит файлы, изменённые до текущей задачи, если пользователь явно не просил их трогать.
4. Если в рабочем дереве есть чужие dirty-файлы, оставить их unstaged и явно отметить это в итоговом сообщении.
5. Если после финальной проверки внесена новая правка, повторить только проверки, относящиеся к изменённому файлу, и затем снова выполнить `git diff --cached --check`.
